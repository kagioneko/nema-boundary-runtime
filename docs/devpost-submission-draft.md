# Devpost Submission Draft — NEMA Boundary Runtime

## Track

Developer Tools

## Tagline

Turn conversational signals into executable, inspectable AI safety policies.

## Inspiration

LLM boundary behavior is often hidden in a static prompt. When a conversation becomes urgent, dependent, or emotionally intense, developers can see the final response but not a deterministic explanation of why control behavior changed. We wanted a small policy layer that could be inspected, replayed, validated, and calibrated independently from the model.

## What it does

NEMA Boundary Runtime separates signal inference from policy execution. A validated `ControlState` enters a deterministic runtime. Versioned policies evaluate that state, emit response-control directives, and produce a condition-level trace. The UI shows signals, fired policies, directives, baseline and controlled responses, a token-level difference view, and the exact execution path.

Developers can temporarily edit a threshold and replay the same input. The override is schema-validated, size-bounded, and never persisted. This makes it obvious that NEMA is an executable developer tool rather than a hidden prompt wrapper.

## How we built it

- FastAPI and Pydantic for strict request, state, and policy validation
- A priority-ordered deterministic policy runtime
- A responsive dependency-free browser UI
- Replayable policy thresholds and execution traces
- A 40-case stratified offline development contract
- Pytest, adversarial body-limit tests, and Playwright desktop/mobile checks
- Docker and one-command local workflows
- GPT-5.6 Sol through OpenRouter for strict Structured Output state inference and baseline/controlled response generation

## Where Codex accelerated the work

Codex helped turn the initial specification into a working runtime, generated adversarial and regression tests, identified policy-schema and request-size edge cases, built the responsive UI and browser harness, and iterated on the evaluation contract. Human decisions fixed the product boundary, scientific disclaimers, privacy defaults, evaluation claims, and publication gates.

## How GPT-5.6 is used

The limited API proof pins `openai/gpt-5.6-sol` through OpenRouter, restricts routing to the OpenAI provider, disables fallbacks, requests data-collection denial, and uses low reasoning effort. State inference uses a strict JSON Schema with seven bounded 0..1 signals plus the `support_escalation` enum. Pydantic validates the returned JSON before the deterministic NEMA runtime evaluates policies. Baseline and controlled responses then use the same pinned model; only the controlled path receives the runtime directives.

The deterministic NEMA runtime—not GPT-5.6—decides which policy fires and creates the execution trace.

## Challenges

The hardest design problem was preserving useful support without turning the system into a cold refusal layer. We also had to distinguish a deterministic conformance suite from model-performance evidence, prevent policy overrides from becoming an arbitrary execution surface, and keep the UI understandable in a short demo.

## Accomplishments

- Same validated state and policy version always produce the same trace
- Invalid fields, operators, directives, duplicate IDs, coercions, and oversized bundles fail closed
- Temporary policy replay never mutates the server policy
- Desktop and mobile browser checks pass without horizontal overflow
- Offline development contract is fully inspectable case by case
- No persistent conversation storage in the MVP

## What we learned

Safety controls are easier to test when model inference, deterministic policy decisions, and response generation are separate layers. Explainability is more useful when it is generated by the runtime itself instead of asking the model to explain its own behavior.

## What's next

- Expand beyond the completed four-scenario / 12-call GPT-5.6 integration proof; it used 3,505 tokens at a reported cost of $0.08235 and is not presented as a benchmark
- Model-specific calibration profiles
- Human review and abstention paths
- Versioned policy signing and organization-level policy packs
- Integration with the wider Emilia boundary protocol

## Important limitations

- ControlState is a behavioral control representation, not a medical or psychological diagnosis.
- The offline 40-case contract is a development conformance suite, not GPT-5.6 performance and not evidence of generalization.
- The prototype does not guarantee prevention of dependency or harm.
